# ADR 006: Интеграция авторизации в KMP/iOS и конфигурация окружений

## Статус
Реализовано

## Контекст
В рамках реализации сквозной авторизации на клиенте необходимо было настроить нативный вход через Google Sign-In в iOS-приложении (Kotlin Multiplatform), безопасно передавать полученные токены на бэкенд (`auth-service`) для обмена на внутренние JWT, а также обеспечить гибкую конфигурацию базового URL бэкенда для разработки (dev) и продакшена (prod).

## Решение

### 1. Клиентская интеграция (KMP & SwiftUI)
*   **KMP Shared Module:** Создан репозиторий `AuthRepository` для выполнения HTTP-запроса обмена токенов (`POST /auth/v1/google`) с использованием Ktor. Успешно полученные JWT (Access и Refresh) сохраняются в защищенном Keychain хранилище устройства с помощью библиотеки `multiplatform-settings`.
*   **SwiftUI Integration:** Логика нативного Google Sign-In и вызова KMP реализована в `AuthViewModel`.
*   **Безопасность потоков (Swift Concurrency):** Все операции изменения состояния интерфейса и вызов Kotlin `suspend` функций изолированы на главном потоке с помощью декоратора `@MainActor` и вызова `Task { @MainActor in ... }`. Это решает ограничение KMP/Native, требующее запуска корутин только на главном потоке при вызове из Swift, и предотвращает краши приложения.

### 2. Поддержка множественных Google Client ID на бэкенде
*   Поскольку нативный iOS клиент генерирует токены с собственной аудиторией (`aud`), отличной от веб-клиента, `auth-service` был доработан для приема списка разрешенных Client ID (как Web, так и iOS) через запятую.
*   Новые реквизиты были успешно добавлены в окружение кластера через HashiCorp Vault.

### 3. Динамическая конфигурация окружений (URL бэкенда)
Для исключения жестко захардкоженных URL-адресов в коде приложения настроена конфигурация через Xcode Build Configurations:
*   В файле `project.pbxproj` заведена пользовательская переменная сборки `BACKEND_URL`:
    *   Для конфигурации **Debug**: `https://dev.simohin.ru`
    *   Для конфигурации **Release**: `https://simohin.ru`
*   В `Info.plist` добавлен ключ `BackendURL` со значением `$(BACKEND_URL)`.
*   Приложение динамически считывает адрес бэкенда при инициализации через `Bundle.main`:
    ```swift
    Bundle.main.object(forInfoDictionaryKey: "BackendURL") as? String
    ```
*   **Сборка из консоли:** При необходимости переопределить адрес бэкенда во время сборки на CI/CD или локально из консоли можно передать параметр напрямую в `xcodebuild`:
    ```bash
    xcodebuild -scheme iosApp -configuration Release BACKEND_URL="https://prod-api.domain.com"
    ```

## Результаты
*   Настроена полностью рабочая цепочка авторизации: клиент получает Google ID Token -> обменивает на бэкенде -> сохраняет JWT -> перенаправляет пользователя.
*   Исключены падения из-за некорректных потоков при вызове Kotlin корутин из Swift.
*   Обеспечен нативный переход на прод-окружение при переключении схемы сборки.
