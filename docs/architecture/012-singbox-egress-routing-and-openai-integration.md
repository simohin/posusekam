# ADR 012: Настройка egress-маршрутизации через sing-box для интеграции с OpenAI

## Статус
Реализовано

## Контекст
В процессе интеграции с OpenAI API (эндпоинт `/v1/ai/generate-products` в `backend-service`) запросы завершались ошибкой `HTTP 403 Forbidden` / `Internal Server Error` из-за ограничений OpenAI на обслуживание запросов из Российской Федерации.

Для обхода блокировки потребовалось настроить проксирование исходящего трафика (egress-routing) на уровне хоста (`simohin-macmini`, Ubuntu 24.04), где запущен кластер `k3s`, с использованием TUN-интерфейса и прокси-протокола Hysteria 2 с обфускацией Salamander.

При реализации возникли следующие технические сложности:
1. **Поддержка транспорта**: Исходный VLESS-профиль использовал Xray-специфичный транспорт `xhttp`, который не поддерживается официальным ядром `sing-box`. Было принято решение переключиться на протокол Hysteria 2.
2. **Конфигурация TLS/uTLS**: Для Hysteria 2 (работающей поверх QUIC/UDP) включение uTLS (`chrome` fingerprint) приводило к ошибке `unsupported usage for uTLS` и падению DNS-резолвера. Настройки uTLS были удалены.
3. **Зацикливание DNS (DNS Loop)**: Системный резолвер `systemd-resolved` перенаправлял все запросы на TUN-интерфейс, но из-за отсутствия явного сопоставления портов в `hijack-dns` запросы уходили обратно в TUN по кругу.
4. **Конфликт с сетью Kubernetes**: Привязка всех сокетов `direct-out` к физическому интерфейсу (`auto_detect_interface: true`) приводила к тому, что внутренний трафик кластера (CoreDNS на `10.43.0.10:53`, Vault на `vault.vault.svc:8200`) перенаправлялся на домашний физический роутер и сбрасывался, ломая запуск подов.

## Решение

Была разработана и применена полностью loop-free конфигурация [sing-box](file:///etc/sing-box/config.json):

### 1. Системные исключения TUN (`route_exclude_address`)
Для предотвращения перехвата внутреннего трафика кластера на уровне ядра Linux в параметры TUN-интерфейса добавлены системные исключения:
```json
"route_exclude_address": [
  "10.42.0.0/16",   // Подсеть подов Kubernetes
  "10.43.0.0/16",   // Подсеть сервисов Kubernetes (включая CoreDNS)
  "192.168.0.0/16", // Локальная сеть хоста
  "172.16.0.0/12",  // Сети мостов Docker
  "100.64.0.0/10",  // Tailscale VPN
  "169.254.0.0/16", // Link-local
  "224.0.0.0/3"     // Multicast / Reserved
]
```
Это позволило внутреннему трафику k3s обходить TUN-интерфейс и идти по стандартным маршрутам ОС.

### 2. DNS-перехват (DNS Hijacking)
В правила маршрутизации (`route.rules`) добавлены правила разделения DNS-запросов:
*   DNS-запросы, направляемые на внутренние адреса Kubernetes (порты `53` в подсетях `10.42.0.0/16` и `10.43.0.0/16`), отправляются напрямую (`direct-out`), минуя перехват. Это сохранило работоспособность CoreDNS Service Discovery.
*   Все остальные DNS-запросы на порт `53` перехватываются действием `"action": "hijack-dns"`.
*   Запросы к доменам `openai.com` и `api.openai.com` резолвятся через прокси-сервер (`dns-proxy` с детором на `hysteria-out`), остальные — через системный DNS (`dns-direct` на `1.1.1.1`).

### 3. Исправление Hysteria 2 Outbound
Из параметров исходящего TLS Hysteria 2 полностью удален некорректный блок `utls` для QUIC-соединений, а также настроен `obfs` Salamander:
```json
"outbounds": [
  {
    "type": "hysteria2",
    "tag": "hysteria-out",
    "server": "finland.simohin.ru",
    "server_port": 23573,
    "password": "uqga9zubupjause0",
    "obfs": {
      "type": "salamander",
      "password": "0cchwwf1zax2u3wr"
    },
    "tls": {
      "enabled": true,
      "server_name": "finland.simohin.ru",
      "alpn": ["h3"]
    }
  }
]
```

## Результаты
1.  **Работоспособность k3s полностью восстановлена**: Init-контейнеры подов (например, `vault-agent-init`) больше не ловят таймаут при резолве `vault.vault.svc` и запускаются мгновенно (время старта пода сократилось до 13 секунд).
2.  **Интеграция с OpenAI успешна**: Запросы из пода `backend-service` к `api.openai.com` автоматически и прозрачно перенаправляются через прокси-туннель.
3.  **Успешное сквозное тестирование**: Вызов эндпоинта `/api/v1/ai/generate-products` с авторизацией JWT-токеном возвращает корректно сгенерированный список продуктов (HTTP 200) без ошибок блокировки.
