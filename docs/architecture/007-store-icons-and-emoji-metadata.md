# ADR 007: Добавление иконок и цветов для магазинов с поддержкой эмодзи и бэкенд-метаданных

## Статус
Реализовано

## Контекст
В приложении требовалось предоставить пользователю возможность выбора визуального оформления для магазинов (сусеков): цвета карточки и иконки. При проектировании возникли следующие ограничения:
1.  **Недостаток системных иконок:** В каталоге SF Symbols отсутствуют подходящие графические изображения для некоторых категорий продуктов (например, для мяса).
2.  **Совместимость версий iOS:** Целевая версия iOS приложения — 16.2. Новые иконки SF Symbols, добавленные в iOS 17 (такие как `donut.fill`, `refrigerator.fill` и др.), не отображаются на более ранних версиях и вызывают пустые области в интерфейсе.
3.  **Локализация и поиск:** Требовалось реализовать локализованный поиск иконок по русским ключевым словам. Зашивать весь каталог метаданных иконок и логику поиска на клиенте непрактично из-за сложности обновления и поддержки.

## Решение

### 1. Перенос метаданных на бэкенд и База Данных
Каталог доступных иконок (как SF Symbols, так и эмодзи) перенесён в базу данных PostgreSQL.
*   Создана таблица `icon_metadata` со следующей структурой:
    *   `id` (UUID, PRIMARY KEY)
    *   `name` (VARCHAR): Идентификатор иконки (имя SF Symbol, например `storefront.fill`, или сам символ эмодзи `🥩`).
    *   `display_name` (VARCHAR): Локализованное название на русском языке (например, "Мясо").
    *   `type` (VARCHAR): Тип ресурса (`SF_SYMBOL` или `EMOJI`).
    *   `category` (VARCHAR): Категория для группировки в UI (например, `FOOD`, `SHOPPING`, `HOUSEHOLD`).
    *   `keywords` (TEXT): Набор ключевых слов для локального и серверного поиска.
*   Написана SQL-миграция Flyway (`V5__Create_Icon_Metadata.sql`) для создания таблицы и её первоначального заполнения ~100 популярными иконками и эмодзи с русским описанием.

### 2. Расширяемый API-эндпоинт метаданных
Реализован read-only эндпоинт `GET /v1/metadata`.
*   Заложена расширяемая структура DTO. API возвращает объект `AppMetadataDto`, содержащий список иконок (`icons`), с возможностью добавления в будущем других настроек (например, палитр цветов `colors` или флагов фичей).
*   На стороне Kotlin Multiplatform (KMP) Shared реализованы кроссплатформенные DTO и [MetadataRepository](file:///Users/t.simokhin/IdeaProjects/posusekam/mobile/shared/src/commonMain/kotlin/ru/simohin/posusekam/metadata/MetadataRepository.kt).

### 3. Условный рендеринг в SwiftUI
Поскольку эмодзи являются текстовыми символами Unicode, а SF Symbols — системными векторными изображениями, в iOS-приложении внедрён условный рендеринг.
*   Создан глобальный Swift-хелпер:
    ```swift
    func isEmoji(_ character: String) -> Bool {
        guard let firstScalar = character.unicodeScalars.first else { return false }
        return firstScalar.properties.isEmoji && (firstScalar.properties.isEmojiPresentation || character.unicodeScalars.count > 1)
    }
    ```
*   В компонентах `StoreCard` (карточка магазина) и `StoreFormSheet` (экран редактирования) настроена отрисовка:
    *   Если `isEmoji(icon)` возвращает `true` $\to$ отображается `Text(icon)` с размером шрифта `.system(size: 24)`.
    *   Иначе $\to$ отображается `Image(systemName: icon)` с белым цветом заливки.

### 4. Интерактивный селектор `IconPickerView`
*   Экран выбора иконок `IconPickerView` переписан под динамическую загрузку данных из `AuthViewModel.metadata`.
*   Внедрен `Picker` (Segmented Control) для переключения между "SF Symbols" и "Эмодзи".
*   Иконки сгруппированы по категориям с заголовками.
*   Добавлен локальный статический список-fallback на случай отсутствия соединения при первом открытии экрана.
*   Поиск фильтрует элементы по названию и ключевым словам, приходящим с сервера.

## Результаты
*   Решена проблема с отсутствием иконки для мяса (добавлены эмодзи 🥩, 🍗, 🐟 и другие продукты).
*   Заменены несовместимые с iOS 16.2 иконки SF Symbols (используются проверенные аналоги из iOS 14+).
*   Создан гибкий бэкенд-механизм метаданных, который позволяет легко обновлять и дополнять каталог иконок и ключевые слова без обновления клиентского приложения.
